### 1. `LogEntry` `card_state`/`history` full-repost redesign not implemented (merged finding — `code-card` + `code-press`)
**Spec:** `specs/object_specs/ipfs_card.md` §5 (amended 2026-07-16)
**Code to implement in:** `press/src/functions/log.ts` (`appendLogEntry`)
**Gap:** the spec requires every post-genesis `LogEntry` to carry a full repost of current state (`card_state`) and a flat, oldest-first `history` array of every predecessor CID, plus `version` = current log length + 1. Current code constructs only the pre-redesign diff-only shape (`field_updates`/`revocation`, no `card_state`, no `history`) and hardcodes `version = 1` with an explicit code comment: `// Increment logic requires full chain walk; placeholder for Phase 3.`
**What's needed:** `appendLogEntry` must fetch/decrypt the current head (genesis `CardDocument` or prior `LogEntry`), populate `card_state` (current field state with `field_updates` applied) and `history` (prior head's `history` array + prior head's own CID appended), and compute `version` as `history.length + 1`.

### a. On-chain write-gate doesn't bind signed payload fields to calldata
**Source:** `code-contracts.md` Findings 1 and 2.
**Issue:** every write operation's authorization check (`contracts/logic-contract/src/write_gate.rs`, used by `register_card`, `update_card_head`, `claim_open_offer`, `batch_update_card_heads`, `register_sub_card`, `deregister_sub_card`, `set_policy_address`, `remove_policy_address`) verifies the secp256r1 signature over the raw signed payload, then only checks two fields extracted from that payload — `op` and `sequence` — against anything. No other field (`card_address`, `policy_address`, CIDs, the `updates` array in batch operations, `domain`/`path` in DNS operations) is ever compared against the corresponding calldata parameter actually being written. Separately, the DNS-admin `AdminAuthorizeSubCardPayload` check (`register_sub_card`, `subcard_ops.rs`) only verifies that `sub_card_address`/`sub_card_doc_cid` fields are *present* in the payload — never that they *equal* the calldata values — despite the spec's E-47 explicitly requiring equality.
**What's at stake:** a signature+payload pair, once observable (mempool, logged/retried client payloads), does not cryptographically bind the signer to a specific card address, CID, or policy. ### l. Verifier code introduces an `app_card_chain_valid` result field the spec doesn't define
**Source:** `code-verifier-sdk.md` Finding 2.
**Issue:** `card_verifier.md` §7.2/§8 folds the app-certification chain-walk outcome entirely into `scope_clean` (a chain-walk failure is a hard rejection, no separate result field). Both language ports add an independently-inspectable `app_card_chain_valid: boolean | "skipped"` field not present in the spec's result shape at all.
**What's at stake:** this is precisely the field tracking whether the Phase-1-hardened defense-in-depth security check passed — its shape (folded into `scope_clean` only, vs. independently queryable) is a real API design decision for a security-relevant result, not an incidental extra field.
**Needs a decision on:** formalize `app_card_chain_valid` in the spec's §8 result type (letting callers distinguish "app-cert chain failed" from other Stage 2 rejections — the reviewing agent's recommendation, as the extra granularity seems useful and low-risk), or remove it from code so callers rely purely on `scope_clean` + the `APP_CARD_CHAIN_NOT_TRUSTED` error code as currently specified.
