## Finding 1 — Overview's privacy claim ("room membership is private") contradicts `matrix_room.md`'s own paraphrase of the same source and its own visibility table

**Conflicting specs:** `room_discovery.md` Overview (line 14) vs. `matrix_room.md §What the Synapse Operator Can See` (line 155) and its visibility table (lines 148–156).

**Description:** `room_discovery.md`'s Overview attributes to `card_protocol_spec.md`'s "general stance" the claim that "only message content **and room membership** are meant to be private, not policy or chain data" (emphasis added). But `matrix_room.md` — citing the identical source ("`card_protocol_spec.md`'s general stance") one paragraph after its own visibility table — states the narrower claim: "policy and chain data are public, **only message content is private**," with no mention of room membership. `matrix_room.md`'s own table one section earlier explicitly lists "Room membership (which Matrix user IDs have joined)" as **visible (✓)** to the Synapse operator — i.e., room membership itself is not private; only the mapping from a Matrix user ID *back to a `card_hash`* is protected (via the one-way commitment scheme in `matrix_encryption.md §3`). `room_discovery.md`'s wording conflates "the `card_hash`-to-Matrix-ID binding is protected" with "room membership is private," which is materially different and reads as a direct contradiction of `matrix_room.md`'s table entry when the two docs are read side by side.

**Recommended resolution:** Reword `room_discovery.md`'s Overview sentence to match `matrix_room.md`'s framing — either drop "and room membership" entirely (matching `matrix_room.md`'s exact phrase), or if the intent was to reference the `card_hash`↔Matrix-ID binding protection specifically, say so explicitly (e.g., "...only message content is private, and the `card_hash` behind a room membership entry, not policy or chain data or the fact of membership itself"). Whichever phrasing is chosen, it should be made to literally match `matrix_room.md`'s paraphrase of the same underlying `card_protocol_spec.md` stance, since both docs claim to be quoting the same source.

## Finding 2 — `wallet_sdk.md`/`app_sdk.md` have no corresponding coverage of the client-side discovery path they're implicitly assigned

**Conflicting specs:** `room_discovery.md §2` and §3's "Not the default. Client SDKs should attempt the local path first..." vs. `wallet_sdk.md` and `app_sdk.md` (both silent).

**Description:** `room_discovery.md` repeatedly assigns responsibility to "client SDKs" for implementing `discoverRooms()` (§2, a pure client-side function) and for the local-first/server-fallback policy (§3: "Client SDKs should attempt the local path first and fall back to this endpoint only when local RPC/IPFS access genuinely isn't available"). Neither `wallet_sdk.md` nor `app_sdk.md` (the two SDK object specs in scope) mentions `discoverRooms`, the room index, `/matrix/discover-rooms`, or any room-discovery capability at all — there is no defined SDK-level function signature, exposed API, or fallback-detection logic corresponding to what `room_discovery.md` assumes exists. This is the same "spec gap" pattern the implementation plan calls out explicitly (§Phase 2 note: "process X has no spec for lifecycle stage Y" is itself a log-worthy gap, not a silent pass) — here it's "process room-discovery assumes a client-SDK capability that no SDK spec defines."

**Recommended resolution:** Either (a) add a short subsection to `wallet_sdk.md` and/or `app_sdk.md` defining the `discoverRooms`-equivalent client API surface (function signature, which SDK owns it, how "local RPC/IPFS access isn't available" is detected to trigger fallback), or (b) if this is intentionally left as an application-level integration concern rather than an SDK-provided primitive, add a line to `room_discovery.md` §2/§3 clarifying that the client-side function is a reference algorithm apps may implement directly rather than a guaranteed SDK-exposed method, so the "client SDKs should..." language doesn't imply an SDK commitment that doesn't exist anywhere else in the spec set.

## Finding 3 (minor) — one-sided companion-document listing between `room_discovery.md` and `matrix_synapse_module.md`

**Conflicting specs:** `room_discovery.md`'s companion-document list (line 6: `matrix_room.md`, `matrix_room_membership.md`, `matrix_join_attestation_and_revocation.md`) vs. `matrix_synapse_module.md`'s companion-document list (line 7: `matrix_room.md`, `matrix_encryption.md`, `matrix_room_membership.md`, `matrix_join_attestation_and_revocation.md`).

**Description:** `room_discovery.md §2` step 3b leans heavily and substantively on `matrix_synapse_module.md`'s `predicates.py` ("a direct reuse of `predicates.py`'s logic, not a reimplementation with different semantics... share this logic (or at minimum be tested against the same fixtures)") — a load-bearing cross-reference, not a passing mention. Yet neither document lists the other as a companion document, and `matrix_synapse_module.md` never mentions `room_discovery.md` or the client-side discovery path anywhere in its own text. This is a much smaller issue than Findings 1–2 (no factual contradiction, just an asymmetric/missing cross-reference), but given how much §2 step 3b depends on `predicates.py`'s exact behavior staying in sync with a client-side reimplementation, the two docs not listing each other as companions — and `matrix_synapse_module.md` giving no acknowledgment that an external client-side consumer depends on its evaluator's exact semantics — is worth tightening.

**Recommended resolution:** Add `specs/process_specs/room_discovery.md` to `matrix_synapse_module.md`'s companion-document list, and add a one-line note near the `predicates.py` package-layout entry acknowledging that its evaluation semantics are also relied on by a client-side reimplementation per `room_discovery.md §2`, so a future edit to `predicates.py`'s behavior is flagged as needing a corresponding client-side check.
