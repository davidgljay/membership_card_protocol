## Findings flagged during inventory (carried into Phase 2's discrepancy log, not resolved here)

1. **`PUT /accounts/{card_hash}/keyring`'s `rotate_service_secret` parameter** is not in `implementation-plan.md` §Step 2.4 at all. Code comments attribute it to "client-sdk implementation plan Step 2.4 fix." Needs verification against `plans/client-sdk/implementation-plan.md` and inclusion in the object spec regardless of which plan owns it.
2. **UUID registration and subcard deregistration now require a signed envelope** proving control of the sub-card private key. Both `strategic-plan.md` (Goal 3 objectives) and `implementation-plan.md` §Steps 5.1/5.2 describe these as requiring "no authentication beyond valid card_hash." Code comments attribute the change to "security-audit finding (a)" and cite `implementation-plan.md §Step 2.7`, which does not exist in `plans/wallet-service/implementation-plan.md` (that document has no Step 2.7 — its Phase 2 steps are 2.1-2.4). This citation likely points to a different plan document (possibly `plans/client-sdk/implementation-plan.md`) and needs to be traced in Phase 2.
3. **An entire OHTTP (Oblivious HTTP) subsystem** (`/ohttp/gateway`, `/ohttp/key-config`, `src/ohttp-gateway.ts`, `src/ohttp-router.ts`) exists and is not mentioned anywhere in `plans/wallet-service/strategic-plan.md` or `implementation-plan.md`. Comments attribute it to "client-sdk implementation plan Step 1.4c." This is a full routing path that dispatches to the same logic as several other endpoints (`accounts-challenge`, `accounts-create`, `keyrings-get`, `messages-create`) and must be documented in the object spec's endpoint list, cross-referenced against `specs/process_specs/oblivious_transport.md`.
4. **`DELETE /messages/{uuid}` now keys off `message_queue.delivery_uuid`**, not the message's own primary key or `card_hash` lookup as the Phase 1 schema sketch implied. Confirm exact column in Step 1.2.
5. **Two federation endpoints (`POST /federation/keyrings`, `POST /federation/keyrings/delete`) log via raw `console.info`**, not the `auditLog()` helper used everywhere else (Step 6.2's structured JSON logging). Same for `messages-create.ts` and `subcard-uuid-registration.ts`/`subcard-deregistration.ts`, which also use `console.info` for their "received"/"registered"/"deregistered" lines. Worth checking in Phase 1 Step 1.4 / Phase 2 whether this is a documentation gap (audit-log-schema.md doesn't cover these events) or a real inconsistency with the "single `auditLog()` helper" claim in `phase-6-summary.md`.
