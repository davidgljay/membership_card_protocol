# Phase 1 Milestone Review — Foundation and Provider Architecture

**Date:** 2026-07-03
**Scope:** This phase touched four codebases — `client-sdk/` (the SDK itself), `relay/` (Step 1.4b), `wallet-service/` (Step 1.4c), and `press/` (Step 1.4d) — building the full oblivious-relay transport end-to-end for both destinations the SDK talks to, alongside the SDK's provider architecture, crypto/canonicalization core, verifier integration, default web/RN providers, and CI.

## Summary

Every provider interface from Step 1.2 (`StorageProvider`, `SecureKeyProvider`, `PasskeyProvider`, `RealtimeTransportProvider`, `ObliviousProtocolTransport`, `MultiInstanceLock`) has both a web and RN default implementation passing the shared Step 1.2 contract suite (150 tests across `client-sdk`/`client-sdk-web`/`client-sdk-rn`); the crypto core matches the verifier package's canonicalization byte-for-byte against the full conformance corpus and round-trips ML-DSA-44/ML-KEM-768/HPKE correctly; the verifier-integration smoke test passes; and the shared cross-platform scenario harness (Step 1.7) runs identically on both platforms. The oblivious-relay transport (Steps 1.4a–1.4d) is implemented and unit/integration-tested end-to-end at the component level — `ObliviousProtocolTransport` against a stub relay+destination, relay's real forwarding endpoint against a real self-signed-HTTPS stub target, and both wallet-service's and press's real gateway dispatch logic (real HPKE, real handler functions, real Postgres for wallet-service) — with auth-parity confirmed by direct test for both (an invalid-signature case compared directly against the gateway dispatch for wallet-service; mocked-handler routing/error-mapping tests for press). CI is configured for `client-sdk/` (`.github/workflows/client-sdk-ci.yml`); `relay/`, `wallet-service/`, and `press/` don't yet have per-service CI workflows in this repo (only `client-sdk-ci.yml`, `wallet-service-ci.yml`, `relay-deploy.yml`, and `publish-verifier.yml` exist), so "CI green on all four workspaces" is verified here by running each workspace's own test/build commands locally rather than by a workflow run.

## What was **not** fully verified, and why

The Milestone Review's stated bar — a live 3-hop round trip (device → real relay → real wallet-service/press gateway → back, against actually-running processes, not stubs) — was attempted directly rather than assumed. It surfaced a **pre-existing, previously-undetected bug**, unrelated to any Phase 1 work: wallet-service's real `node-server` build (Nitro's bundled `.output/server/index.mjs`) throws `ERR_PACKAGE_PATH_NOT_EXPORTED` for `@noble/hashes/sha3` on requests that transitively touch both `viem`/`ox` (e.g. `keccak256OfBase64Url`) and `@noble/post-quantum` (e.g. `verifyMasterCardSignature`) — which is most routes, including the pre-existing `/accounts/challenge`. Confirmed via direct reproduction against commit `85bb3eea` (before any of this phase's work), so it predates this plan entirely; the existing vitest suite never catches it because vitest resolves modules from source (each package correctly finds its own isolated `@noble/hashes` copy), while Nitro's bundler collapses two incompatible coexisting versions (`@noble/post-quantum` → `@noble/curves@2.0.1` → `@noble/hashes@2.0.1`, vs. `viem` → `@noble/curves@1.9.1` → `@noble/hashes@1.8.0`) into one in the bundled output. This blocks running wallet-service's *real production build* at all, not just the oblivious-relay path specifically — flagged separately as its own task rather than fixed here, since it's outside this plan's scope and touches wallet-service's core dependency graph.

Given that, the full live round trip was validated in the pieces that *are* independently exercisable:
- `ObliviousProtocolTransport` → relay's real forwarding endpoint, using a real self-signed HTTPS stub destination (not wallet-service itself, to sidestep the build bug) — passes, confirming relay correctly forwards an opaque HPKE blob byte-for-byte and returns the response byte-for-byte, for both a wallet-service-shaped and a press-shaped target, with an unknown `target_id` rejected before any forward (`relay/server/integration-tests/ohttp-forwarding.test.ts`).
- Wallet-service's real gateway logic (`decapsulate` → `dispatch` → `encapsulateResponse`), run directly under vitest against a real Postgres instance with real HPKE sealing/opening (`test/ohttp-gateway.test.ts`, `test/ohttp-router.test.ts`) — these call the actual functions in-process (real crypto, real DB), not over real HTTP, and not through Nitro's bundler, so they don't exercise the bundled build's broken module resolution. This confirms the protocol logic itself is correct; it does not by itself confirm the real HTTP route wiring (`server/routes/ohttp/*.ts`) works end-to-end, since that requires the currently-broken bundled build (or an equivalently real dev-server run, which wasn't attempted here) to actually serve a request.
- Press's gateway dispatch logic, unit-tested with the real six handlers mocked (since exercising the real ones needs a full `PressContext` this phase didn't need to construct) plus a real HPKE round-trip test.

A genuine, fully-live three-process round trip (spawning real relay + real wallet-service + real press `node-server` builds simultaneously, wired together over real network calls, no stubs anywhere) remains open — this is explicitly Step 6.2's job ("Oblivious-relay transport validation against real relay, wallet-service, and press OHTTP endpoints," in Phase 6, after the wallet-service build bug above is fixed), not a Phase 1 gap being silently carried forward.

## Other notes

- `press/src/context.ts`'s `buildCardVerifier()` doesn't pass `appCertificationRoot` to the `CardVerifier` constructor, which the verifier package's current code requires (throws `MISSING_APP_CERTIFICATION_ROOT` otherwise) — a pre-existing bug noticed in Step 1.4, unrelated to this plan and not fixed here.
- `@membership-card-protocol/verifier` and `-ipfs-provider` are not published to npm yet (`file:` dependencies, with a `// TODO` marker matching `press/package.json`'s existing convention); `client-sdk`'s exports maps needed an explicit `"require"` condition added for Jest's CJS-style resolver, which doesn't exist for either.
- Steps 1.4b/c/d each deviate from the plan text's literal `Content-Type: message/ohttp-req` in the same way: the wire format is plain JSON (`{ enc, ciphertext }`), matching CP-0's resolved lightweight-envelope design, not RFC 9458 Binary HTTP framing.
