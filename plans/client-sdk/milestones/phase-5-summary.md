# Phase 5 Milestone Review — Messaging and UUID/Relay Management

**Date:** 2026-07-04
**Scope:** `client-sdk/packages/client-sdk/src/messaging/` (Steps 5.1–5.6), plus one end-to-end scenario test added to each of `client-sdk-web` and `client-sdk-rn` for Step 5.5's platform-parity requirement.

## Summary

`envelope.ts` (5.1) implements `CardMessageEnvelope` construction for this SDK's in-scope message-type taxonomy (`text`, `reply`, `edit`, `reaction`, `read_receipt`, `card_offer`/`card_offer_accepted`/`card_offer_declined`, `card_update_notification`, `auth_request`/`auth_response`), with `edit_of`/`retracts`/`forwards` mutual exclusivity enforced at construction time. `fanout.ts` implements sender-side per-subcard fan-out (`message_routing.md §Sender-Side Fan-out`): independent ML-KEM-768 encryption to each of a recipient's registered sub-cards, producing N distinct routing envelopes. `inbound.ts` (5.2) decrypts and verifies inbound envelopes via the shared `CardVerifier.verifyEnvelope()` — never a hand-rolled check — and deduplicates by message ID against a `StorageProvider`-backed history. `uuidRegistration.ts` (5.3) implements wallet-facing UUID registration with structural session-per-card separation (a transport-factory pattern, not a shared transport instance) and randomized staggering between cards. `replenishment.ts` (5.4) schedules proactive UUID pool replenishment that is never triggered in the same tick as a message receipt, via an injectable-schedule design that makes "replenish synchronously on receipt" structurally unrepresentable. `delivery.ts` (5.5) covers all three realtime delivery paths (SSE, WebSocket, silent-push-triggered `GET /pending`) with `ack()` as the sole path to relay clearance — no delivery function ever calls it. `uuidDeregistration.ts` (5.6) implements signed-envelope-authenticated wallet-service-local deregistration, structurally and operationally distinct from on-chain sub-card revocation.

A two-simulated-device end-to-end test (`test/messaging/phase5Milestone.test.ts`) confirms independent per-subcard message delivery: one holder's card registered on two devices, each with its own sub-card ML-KEM keypair, both correctly and independently receive, decrypt, and verify a single fanned-out message, with cross-decryption using the wrong device's key confirmed to fail, deregistering one device's UUID pool confirmed to leave the other device's delivery unaffected, and a second message after that deregistration confirmed to still reach the still-registered device correctly. Step 5.3's own session-separation and staggering tests (`test/messaging/uuidRegistration.test.ts`) continue to pass unmodified.

243 tests pass in the `client-sdk` core package (up from 206 at the start of this phase, +37 across Steps 5.1–5.6 and the milestone's own end-to-end test); 25 in `client-sdk-web` (up from 24, the new Step 5.5 platform scenario); 22 in `client-sdk-rn` (up from 21, same); build/typecheck/lint clean across the whole workspace.

## Findings from implementation, not anticipated by the plan text

**1. `CardVerifier.verifyEnvelope()`'s `SignedMessageEnvelope` type is a generic signed-document shape, not this protocol's messaging envelope — and the two happen to be runtime-compatible only because the type's `message` field constraint is never actually read.** The plan's Step 5.2 instructs verifying "via the shared `CardVerifier`'s `verifyEnvelope()`," implying a natural type fit. In practice, the verifier package's own `SignedMessageEnvelope` (`{ payload: { message: string; protocol_version: string; timestamp: string; [key: string]: unknown }, signatures }`) is a different, more generic interface than this SDK's `CardMessageEnvelope` (`{ payload: MessagePayload, signatures }`, no `message` field at all) — the two even share a name at first, forcing a rename to `CardMessageEnvelope` to avoid an ambiguous re-export collision at the package barrel. Tracing `verifyStage1`/`extractProtocolVersion` in the verifier package's source confirmed that `message` is a type-level-only requirement never read at runtime — the verifier canonicalizes and signs over `payload` exactly as received, and only reads `protocol_version` and `timestamp` from it. This meant a `CardMessageEnvelope` round-trips through `verifyEnvelope()` correctly with a type-level cast only, no runtime transform, no field renaming, and no duplicated verification logic — but this was confirmed by reading the verifier's actual implementation, not something the type signatures alone guaranteed. **Resolution:** documented explicitly in `inbound.ts`'s doc comment and in `client_sdk.md §10.2`, flagged as the one place that would need to change if a future verifier version starts reading `payload.message` at runtime.

**2. There is no on-chain "list sub-cards for card" RPC method — sender-side fan-out necessarily depends on caller-supplied resolution, matching (not deviating from) established precedent.** `message_routing.md §Sender-Side Fan-out` reads as though resolving "the recipient card's current sub-card list from the on-chain storage contract" is a simple, uniform lookup. The verifier package's `RpcProvider` interface has no such method — the "active sub-card list" the spec refers to lives inside the *decrypted* master `CardDocument`, a policy-defined field the verifier package treats as opaque (`[key: string]: unknown`) since its own chain-walk/revocation logic never needs to enumerate it. `fanOutMessageToSubCards` therefore takes the resolved `SubCardRecipient[]` list as a direct parameter rather than resolving it itself — the same category of gap this SDK has already documented for `policyAddress` resolution (§8.2) and app-identity resolution (§9.3), not a new one. No code change was needed to accommodate this; it only needed to be named explicitly rather than silently assumed away, since a caller reading only the plan's prose might expect this SDK to resolve the list on its own.

**3. A real signing/verification-shape bug, caught while writing Step 5.6's registration/deregistration interop test, not by Step 5.3's own test suite.** `registerCardUuids` (Step 5.3) originally embedded `public_key` as a field *inside* the signed `payload` object in the wire body, but never included it in the bytes actually passed to `sign()` — the signature covered a payload shape without `public_key`, while the wire body's `payload` field had it added afterward. Step 5.3's own tests never caught this because they used a stub transport that didn't independently re-verify the signature against the wire body. Writing Step 5.6's stub-wallet-service test (which *does* independently re-verify, mirroring what a real wallet service does per `§Process 1` step 7) surfaced the mismatch immediately. **Resolution:** moved `public_key` to a sibling field of `payload` in the request body, so the exact bytes that were signed and the exact bytes a verifier canonicalizes are always identical — the wallet service already independently re-derives `keccak256(subcard_pubkey) == subcard_hash` and resolves the subcard's public key from on-chain data per the spec, so `public_key` was only ever meant as a convenience field, never load-bearing for verification, but the original code accidentally made it part of the signed content anyway. This is exactly the kind of latent defect that only becomes visible once a caller performs the same signature check the real counterparty will — a lesson mirrored back into this phase's own workflow: Step 5.6's test doubles as an integration check on Step 5.3, not just a unit test of deregistration in isolation.

## What was **not** built in this phase (explicitly out of scope, not gaps)

- `api.*`, `mcp.*`, `introduction`, `announcement`, `delete`, `flag`, and `error` message types (`messaging_protocol.md`'s full taxonomy) — `MessageType` is a literal union covering only this SDK's in-scope subset per the plan's own "at minimum" framing; the union type makes constructing an out-of-scope envelope a compile error, not an unenforced convention.
- One-time prekeys / forward secrecy for message encryption (`messaging_protocol.md` MSG-OQ-3a) — explicitly framed by that spec as a P2 consideration, not a Phase 5 requirement.
- Tor or Nym-based sender/wallet-service anonymity beyond the oblivious-relay transport already built in Phase 1 (`message_routing.md §Transport Extensibility`'s `0x04` flag) — out of this SDK's scope; the oblivious-relay path (Step 1.4a) already provides device-IP hiding, and this phase's own job (session separation + staggering) is a complementary, not overlapping, protection.
- A real relay `/register` (bootstrap/replenishment) HTTP client bundled as a default — `relayFetch`/`relayHttp` are caller-supplied minimal interfaces throughout this phase's modules (`uuidRegistration.ts`, `delivery.ts`), matching the established pattern of not bundling a specific HTTP client version beyond what a provider needs.
- Any UI/consent surface for message display, read receipts, or flag/report flows — this SDK exposes only the protocol-level primitives; display and moderation UI are host-app concerns, consistent with every prior phase's scope boundary.
