## Checklist (per implementation plan's Phase 5 Milestone Review)

- **Does the chosen crypto binding actually build and run on all three targets, not just the one it was prototyped on?** **No — and this is expected, not a miss.** Web: yes, confirmed with a real end-to-end Megolm round trip. `client-sdk` (shared interfaces): yes, clean typecheck and passing tests. **React Native: does not build or run** — `UnimplementedRNMegolmCryptoProvider` throws on every method by design. This isn't a shortfall in this phase's execution; it's the direct, honest consequence of David's Step 17 decision to build a real native crypto module rather than adopt an existing one, and that build (Rust toolchain, Xcode, Android Studio, `uniffi-bindgen-react-native`) genuinely cannot happen in this sandbox. The RN stub documents exactly what remains for whoever picks it up.
- **Is signature verification actually enforced on decrypt (not just present in code but unreachable/unused)?** Confirmed — re-read the actual test asserting `SenderBindingMismatchError` is thrown and surfaced to the caller (not swallowed), with the specific `signerCardHash`/`matrixSender` fields populated correctly.
- **Are there any plaintext leaks — does anything log the decrypted message body anywhere in the stack?** Checked directly: zero `console.log`/`console.debug`/`console.info` calls anywhere in the new Matrix crypto code (`client-sdk/src/matrix/`, `client-sdk-web/src/MegolmCryptoProvider.ts`, `client-sdk-rn/src/MegolmCryptoProvider.ts`). Not checked in this pass: Synapse's or `wallet-service`'s own logs, since neither runs live in this sandbox — this is an honest gap in this specific review, not a claim of full verification across the whole stack.
- **Does joining without a valid attestation actually fail end-to-end, and does `discoverRooms` work from a real client-sdk build?** **Partially confirmed.** Step 18's join-attestation test confirms the attestation is genuinely attached to the join request body and that a missing attestation is distinguishable by a mock policy-enforcing server — but this is tested against a mock server, not a live Synapse + `matrix-policy-module` instance, since none is running in this sandbox (the same Docker-unavailable limitation noted throughout Phases 3/4). `discoverRooms` (Phase 4, fixed same-day for the `verifyCard`/`verifyEnvelope` bug) already runs from the real `client-sdk` build with a real `CardVerifier` end-to-end test — that part of the criterion is genuinely met, just not from *this* phase's new code.
