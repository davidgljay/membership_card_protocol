## What was built

**Step 5 + 5c — Compose stack** (delegated to Haiku, one issue found and fixed on review)
`wallet-service/docker-compose.yml` extended with `synapse-postgres` and `synapse` services, `synapse_pg_data`/`synapse_media`/`synapse_membership_registry` volumes, and a `card-net` network. `wallet-service/matrix/Dockerfile` (`FROM matrixdotorg/synapse:latest` + `COPY matrix-policy-module/`). `.env.example` extended with `MATRIX_SERVER_NAME`, `ARBITRUM_RPC_WS_URL`, `JOIN_ATTESTATION_FRESHNESS_SECONDS`, `WATCHER_BACKSTOP_INTERVAL_SECONDS`, `MATRIX_MEMBERSHIP_REGISTRY_PATH`. **Caught on review:** the agent had added `networks: [card-net]` to the pre-existing `postgres` service — an unnecessary modification (nothing connects wallet-service's own Postgres to Synapse) that violated the step's own "don't modify existing services" done-when criterion. Reverted.

**Steps 7 + 7b + 7c — Matrix credential generation** (Sonnet; **failed mid-task on an account-level session usage limit**, not a bug — assessed and confirmed before continuing, see Notes)
`server/db/migrations/1772400800000_matrix-credentials.cjs` (a `matrix_credentials` audit table mirroring the existing `accounts.service_secret_enc` envelope-encryption pattern), `src/matrix-credentials.ts` (`recordMatrixCredential`/`decryptMatrixCredential`/`listMatrixCredentials`), and `scripts/generate-matrix-secrets.ts`, which generates all four credentials (Synapse's Ed25519 signing key, its `registration_shared_secret`, the watcher bot's login password, and the membership registry's AES-256 key). Real design work, not just mechanical generation: `SecretsService.encryptSecret`/`decryptSecret` is built for secrets *wallet-service itself* later decrypts, but all four of these are consumed directly by the `synapse` container / Python module at its own startup — a different runtime with no way to call wallet-service's API to unwrap a DEK. Resolution, applied uniformly: raw key material is written to a gitignored, volume-mounted file the consuming process reads directly; an encrypted audit-trail copy is *separately* kept via the normal `SecretsService` flow for recovery/rotation bookkeeping only, never on the runtime boot path. Also independently verified (web search, not assumed): Synapse's admin API has no dedicated "kick a user from a room" endpoint — the design uses a dedicated bot account with room-level power (via `power_level_content_override`) and the ordinary Client-Server kick call, rather than a full server-admin token, matching the plan's own least-privilege ask.

**Follow-up fixes after the failed agent's work (done directly, not re-delegated):** added the missing `.gitignore` entry for `matrix/secrets/`; resolved the `registration_shared_secret` inlining question the script had left open (see Step 6) by adding a pre-wrapped `registration-shared-secret.yaml` output, verified against Synapse's actual Docker entrypoint behavior rather than assumed.

**Step 6 — `homeserver.yaml`** (delegated to Haiku after resolving two judgment calls myself first; **one real bug found and fixed on review, plus one root-cause spec error discovered and corrected**)

Two judgment calls resolved before delegating (making the remaining work genuinely mechanical, appropriate for Haiku): (a) `registration_shared_secret` inlining via a second Synapse config file rather than an entrypoint wrapper, and (b) reconciling `matrix_synapse_module.md`'s vague "obtained via the secrets abstraction" language for the membership-registry key with the concrete mounted-file mechanism actually built for the other three credentials — added an explicit `membership_registry_key_path` config key to match reality.

**Bug caught on review:** the Haiku agent's `command:` override for the `synapse` service was `["python", "-m", "synapse.app.homeserver", "--config-path=...", ...]` — but the base image's `ENTRYPOINT` is `/start.py` (unchanged by our Dockerfile), and `command:` supplies *arguments to that entrypoint*, not a replacement for it. `start.py` expects a mode keyword (`run`, `generate`, ...) as its first argument, not `"python"`. This would very likely have failed at container start. Fixed to the confirmed-correct form: `["run", "--config-path=/data/homeserver.yaml", "--config-path=/data/secrets/registration-shared-secret.yaml"]`.

**More serious finding, root-caused during this step's execution:** `matrix_synapse_module.md` (written during Phase 1) claimed "Synapse substitutes `${VAR}` from the process environment when parsing `homeserver.yaml` (standard Synapse config templating)." **This is false** — verified against Synapse's own docs and issue tracker; general `${VAR}` substitution in arbitrary config keys has never shipped in Synapse (it's a long-open feature request, e.g. `matrix-org/synapse#11489`, `#7758`), and Docker Compose's own `${VAR}` resolution doesn't reach into a bind-mounted file's contents either. As written, every templated value in `homeserver.yaml` — including the entire `modules:` block the whole policy-enforcement mechanism depends on — would have loaded into Synapse as literal, unresolved strings. **Fixed:** `matrix/homeserver.yaml` is now `matrix/homeserver.yaml.template` (committed, with `${VAR}` placeholders); a new script, `scripts/render-matrix-config.ts`, renders the real, concrete, gitignored `matrix/homeserver.yaml` by substituting `process.env` values directly, failing fast (same convention as `src/config.ts`) on any missing variable. Tested both paths directly (success with dummy values, and the fail-fast path on a missing variable) — both work correctly. `matrix_synapse_module.md` and the implementation plan's Step 6 are corrected to describe the actual mechanism, not the false claim.

## Verification

Initially only structurally validated (no Docker daemon available). **The user then started a Docker daemon in this environment, enabling a full live-boot test**, which surfaced two further real bugs beyond what static review had caught — both fixed and re-verified:

- `docker compose config` validates cleanly throughout.
- `npx tsc --noEmit` passes for all new/modified TypeScript.
- Ran the actual sequence end-to-end with a local Postgres and a disposable, dummy-valued `.env` (not committed): `docker compose up -d postgres` → `npm run migrate up` (the new `matrix_credentials` migration applied cleanly, alongside two unrelated pending migrations) → `scripts/generate-matrix-secrets.ts` (all four credentials generated correctly: signing key in the right `ed25519 <id> <seed>` format, registration secret both as plain text and pre-wrapped YAML, watcher bot credential, 32-byte registry key — each with a matching encrypted audit row in `matrix_credentials`) → `scripts/render-matrix-config.ts` (rendered `homeserver.yaml` correctly from the template).
- **Two more real bugs found only by actually attempting the build**, neither visible from static review:
  1. `matrix/Dockerfile`'s `COPY matrix-policy-module/ ...` could never have worked with `build: ./matrix` as the compose service's build context — `matrix-policy-module/` lives as a sibling of `matrix/`, and Docker's `COPY` cannot reach outside its build context. Fixed by changing the service to `build: { context: ., dockerfile: matrix/Dockerfile }`.
  2. `docker-compose.yml` *also* had a runtime bind-mount for the same path (`./matrix-policy-module:/data/matrix-policy-module:ro`), directly contradicting the spec's stated design intent that the module is pinned into the image at build time, not fetched at runtime. Removed — a bind mount would have silently shadowed whatever got baked into the image, defeating the whole point of image-pinning.
  - Also added a missing `wallet-service/.dockerignore` — the corrected build context is now the whole `wallet-service/` root (258MB, mostly `node_modules`), which was being uploaded to the build process wholesale and risked `matrix/secrets/` ending up reachable from the build cache.
- **Final live boot result:** `docker compose up -d synapse` built and started successfully. Synapse's logs show it invoked with the correct args (`-m synapse.app.homeserver --config-path=/data/homeserver.yaml --config-path=/data/secrets/registration-shared-secret.yaml`, confirming the earlier `command:` fix), successfully parsed and merged both config files with **no config-parse error of any kind**, and failed only at the last step — `ModuleNotFoundError: No module named 'matrix_policy_module'` — because that Python package doesn't exist until Phase 3, Step 8. This is exactly the anticipated, benign failure mode the plan called out ("the module itself doesn't need to exist yet for this step"). Step 6's actual done-when criterion is now genuinely satisfied, not just structurally approximated.
- A placeholder `wallet-service/matrix-policy-module/.gitkeep` was added, clearly labeled as Phase 2 validation scaffolding only, to be replaced by Phase 3 Step 8's real package.
- Test containers stopped (not removed) after validation; the dummy `.env` used for this test is gitignored and not meant for real deployment.
