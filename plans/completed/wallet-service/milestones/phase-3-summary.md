# Phase 3 Milestone Summary — Recovery Infrastructure (Backup Service)

**Date:** 2026-06-30
**Status:** Complete

Phase 3 implements backup registration, the 72-hour cancellation window, multi-channel notification dispatch, cancellation handling, and key release. All five steps from `implementation-plan.md §Phase 3` are implemented and verified end-to-end against a live local Postgres.

- **Step 3.0 (schema):** `server/db/migrations/1772400200000_phase3-recovery.cjs` adds `keyring_id` to `backup_registrations` and creates `notification_jobs` (Postgres-backed job queue, not Redis, per the plan's durability requirement). Verified against a fresh DB alongside both prior migrations.
- **Step 3.1 (backup registration):** `POST /accounts/{card_hash}/backups` + `GET /accounts/{card_hash}/backups/{backup_id}` (session-token authenticated). `wrapped_blob` is stored and never decrypted, and the GET endpoint omits it from the response entirely. Verified live.
- **Step 3.2 (recovery initiation):** `POST /accounts/{card_hash}/recovery` — no session token (the initiator may not have their device). Creates a `recovery_windows` row with a server-computed `expires_at = now() + 72 hours` (never client-supplied), and a second initiation while one is active returns 409 with the existing `recovery_id`. Verified live.
- **Step 3.3 (notification dispatch):** `src/notifications/` (providers, templates, dispatch) + `server/utils/notification-fanout.ts` (enqueue + one immediate inline attempt per channel) + `server/tasks/sweep-notification-retries.ts` (scheduled retry with exponential backoff, registered via `nitro.config.ts`'s `scheduledTasks`). Default providers are SendGrid (email) and Twilio (SMS), via plain `fetch` calls; both fall back to a console-logging provider when unconfigured, so local dev needs no external accounts. Verified live end-to-end, including the retry path: a deliberately-broken webhook URL failed its inline attempt, was left `pending` with backoff, and the scheduled sweep task picked it up and rescheduled it correctly (confirmed via `curl -X POST /_nitro/tasks/sweep-notification-retries` and inspecting `notification_jobs.attempts`/`next_attempt_at`).
- **Step 3.4 (cancellation):** `POST /recovery/{recovery_id}/cancel` — cancellation credential is the master card key (OQ-WS-6); `challenge` is the recovery_id's own bytes, which both proves the signature targets *this* window (no cross-window replay) and lets the server validate it without any extra state. Verified live: valid signature cancels and fires confirmation notifications; invalid signature → 401; re-cancelling an already-cancelled window → 200 idempotent; cancelling after the 72-hour window has elapsed (even if never released) → 410.
- **Step 3.5 (key release):** `GET /recovery/{recovery_id}/release`. Verified live: before 72 hours → 425 with a `Retry-After` header (confirmed exactly 259200 seconds = 72h); after expiry → releases `wrapped_blob` + `keyring_id` (and only those — no `service_secret`, per the spec); a second release call is idempotent and returns identical data; releasing a cancelled window → 410.

63 automated tests pass (52 from Phases 1-2 + 11 new: `test/notifications.test.ts` for dispatch/template logic with fake providers, `test/backups-repo.test.ts` and `test/recovery-repo.test.ts` for the DB layer including atomicity of cancel/release races), lint and typecheck are clean, and both build presets succeed. The full recovery flow (registration → initiation → notification fan-out → early-release rejection → invalid-cancel rejection → valid cancel → idempotent re-cancel → release-of-cancelled rejection → a second window's full expire-and-release path → idempotent re-release → cancel-after-expiry-without-release) was exercised live against a real Postgres instance, manipulating `expires_at` directly in the DB to simulate the 72-hour wait.

## Security properties verified against the spec's table

Cross-checked `wallet_backup_and_recovery.md §Key Security Properties` against this implementation:

- **Primary/backup service cannot read the wallet:** `wrapped_blob` is stored and returned as opaque ciphertext everywhere — no decrypt is ever attempted on it (grep-confirmed: the only `decryptSecret` calls in the codebase are on `service_secret`, an entirely separate value).
- **Account compromise alone does not decrypt the wallet:** `GET /recovery/{id}/release` returns only `wrapped_blob` and `keyring_id` — never `service_secret`. The holder still needs to complete Phase 2 Step 2.4's re-registration (a separate master-card-key-signed flow) to get a usable `service_secret`.
- **YubiKey/account-compromise theft cannot immediately compromise the wallet:** the 72-hour window is enforced via `expires_at`, a value computed server-side at creation (`now() + make_interval(hours => 72)`) and never accepted from the client at any point in the flow — confirmed by reading every route that touches `recovery_windows`.
- CP-2's three specific concerns are all satisfied: `expires_at` is server-side only; `wrapped_blob` never appears in a `console.info` call anywhere in the codebase (only `card_hash`, `recovery_id`, `backup_id`, and `type`/`channel` are logged); the cancellation challenge is bound to one specific `recovery_id` by construction, so a signature captured for one window cannot cancel another.
- The remaining rows in the security properties table (synced-passkey-requires-account-access, lost-passkey-plus-lost-device, master-key-cold-during-routine-ops) are device/client-side properties with no corresponding backend behavior to verify — noted, not applicable here.

## Bugs found and fixed during this phase

- **Scheduled tasks were silently never registering.** `server/tasks/sweep-notification-retries.ts` was correctly written and correctly listed in `nitro.config.ts`'s `scheduledTasks`, but Nitro logged `Scheduled task sweep-notification-retries is not defined!` and the manual task-invoke endpoint 404'd. Root cause: Nitro's task system requires `experimental: { tasks: true }` in `nitro.config.ts` — without it, `scheduledTasks` entries are accepted by the config schema but never wired up. Added the flag; confirmed live via `POST /_nitro/tasks/sweep-notification-retries`. This would have silently broken every retry in production (Phase 1's `nitro.config.ts` left `scheduledTasks: {}` empty, so this gap was invisible until a real task was registered here).

## Deviations from the plan as written

- The plan's `notification_jobs` polling interval is described as "every 30 seconds." Given the serverless architecture decision from Phase 1 (Cloudflare Workers, no persistent process), continuous 30-second polling isn't available — Cloudflare Cron Triggers and Nitro's `scheduledTasks` both have effectively a 1-minute floor. Implemented instead as: an immediate inline dispatch attempt at enqueue time (covers the common case — all channels typically notified within the same request, well inside the spec's "within 60 seconds" goal) plus a 5-minute scheduled sweep for retries with exponential backoff. This was a deliberate architectural call, not an oversight — flagging it here since it changes the latency characteristics of the *retry* path specifically (first retry could take up to 5 minutes to be picked up, not 30 seconds), though the common-case (first-attempt-succeeds) path is unaffected.
- Added two notification channels not explicitly named in the plan's `notification_jobs.channel` enum sketch: `secondary_contact_email` and `secondary_contact_sms`, splitting the spec's single `secondary_contact` concept by delivery mechanism. The wire format (`notification_channels.secondary_contact: { name, email?, sms? }`) is unchanged from Step 3.1's spec; this only affects internal job-queue bookkeeping.

## Known gaps carried into Phase 4+

- Federation replication of keyring blobs (Step 4.1a) still isn't implemented — recovery's `GET /keyrings/{keyring_id}` fetch-from-any-federation-member step (spec Process 2a Step 7) has no endpoint yet. Single-instance recovery is fully functional; cross-instance recovery is explicitly Phase 4 scope.
- No backup-registration revocation endpoint exists yet. The spec's Process 3 Step 13 ("revoke the old backup registrations at the backup service") during post-recovery re-registration isn't wired up — Phase 2's Step 2.4 keyring rotation doesn't touch `backup_registrations` at all. An old backup registration remains valid (and its `keyring_id` now points at a superseded, soon-to-be-deleted keyring blob) after rotation. Worth flagging for Phase 4 or a dedicated follow-up, since it's a real gap between spec and implementation, not just an unimplemented nice-to-have.
