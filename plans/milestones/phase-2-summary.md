# Phase 2 Milestone Summary — Key Compromise Scenarios

**Date:** 2026-05-22  
**Full report:** `red_teaming/phase_2_report.md`

Phase 2 assessed five key tiers in the trust hierarchy. The highest-severity findings are:

**1. Press sub-card key — Finding 2.2-A (High):** A compromised press key can post backdated 9xx revocations with `notify_holder: false` against any card in its policy scope. The spec explicitly supports this mechanism as a design feature. There is no technical counter in the current protocol. Under ADR-011, revoking a compromised press key requires governance quorum, adding incident response latency. This is the most direct path to weaponizing the protocol against its intended beneficiaries and triggers the Phase 2 clarification checkpoint.

**2. Auditor key — Findings 2.4-A and 2.4-B (both High, partially mitigated by spec change 2026-05-22):** The original design had no forward secrecy — a compromised auditor key decrypted all historical issuance records. The spec now uses an epoch-based audit encryption model: each epoch has a single Audit Encryption Key (AEK) wrapped under each auditor's ML-KEM public key; at epoch close, the auditor produces a signed `AuditEpochCommitment` (entry count + hash commitment over all entry CIDs + findings) and destroys the AEK. Entries from closed epochs are permanently undecryptable. Residual exposure: a compromised current key exposes the current open epoch's records (bounded by epoch duration, annual by default). The audit log remains a high-value state-actor target, but the blast radius is now epoch-scoped rather than all-time.

**3. Policy authorizer key — Finding 2.1-A (High):** Two attack paths remain open after spec changes; one class is now mitigated: (a) **(mitigated)** field definition changes and regex tightening can no longer retroactively invalidate existing cards — the spec now requires compliance checks to use the `policy_id` CID snapshot from issuance, not the current live policy; (b) updating `revocation_permissions` (which evaluates against the current live policy as an explicit carve-out) then submitting mass 9xx intents via the press still brings the full trust chain to a halt. The 9xx path is detectable — a flood of evidenceless revocations is suspicious to any human reviewing them — and the protocol's successor card mechanism enables chain reconstruction (holders present their revoked card to receive a clean successor under a new policy). Recovery is feasible but operationally costly. ADR-011 substantially limited the authorizer key's blast radius by moving press authorization to governance quorum, but the `revocation_permissions` escalation path remains open.

**Finding 2.2-B — governance response latency (High, accepted):** The author confirmed this tradeoff is intentional. A compromised press can spam the chain with easily detectable fraudulent cards (no forged holder signatures) but cannot issue cards that will survive scrutiny. The governance latency window is real but does not enable unrecoverable harm; no protocol change required.

**Spec change (2026-05-22):** Three sections of `specs/card_protocol_spec.md` updated to anchor compliance verification to the `policy_id` CID snapshot at issuance: the Protocol-Required Fields section, §7 verification step 5, §7 step 5a, and the §2 acceptance criterion. The `update_policy` / `revocation_permissions` carve-out (current live policy governs present-time operations) is explicitly documented.

**Cross-phase links to Phase 3:** Finding 2.2-A directly informs the state-actor retroactive de-platforming scenario (Step 3.1-B2) and the intimate-partner revocation-as-harassment scenario (Step 3.3-B). Finding 2.4-A informs the state-actor audit log surveillance scenario (Step 3.1-A1). Phase 3 reviewers should not re-derive these threat models independently.
