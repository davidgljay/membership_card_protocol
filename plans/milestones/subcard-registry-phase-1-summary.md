# Sub-Card Registry — Phase 1 Milestone Summary

**Date:** 2026-07-05
**Status:** Phase 1 (spec & process-spec updates) complete, pending David's sign-off.

Phase 1 of `plans/subcard-registry-implementation-plan.md` is done. Across all touched documents, `active_subcards` is consistently defined as a flat array of base64url ML-DSA-44 public keys (1312 bytes each), protocol-reserved (`protocol-objects.md §1.1`), not present at genesis, mutated only via codes **510** (addition), **511** (removal), and **512** (atomic key rotation) — all three hardcoded to `{ "is_holder": true }` authorization, explicitly not overridable by any policy's `field_definitions` or `update_policy`. This scoping (510/511/512 apply only to entries on the master card's own log, never a sub-card's log, which remains governed by the pre-existing 5xx prohibition in `subcard_creation_policy.md`) is stated identically in `protocol-objects.md §1.1`, `update_codes.md §5xx`, `card_updates.md`, `card_validation.md`, and `subcard_creation_policy.md`.

The previously-undefined "active sub-card list" forward reference in `protocol-objects.md §16`'s verifier chain-walk step 8 is now concrete everywhere it appeared: `protocol-objects.md §16`, `card_validation.md` Stage 2 step 9, `card_verifier.md` §7.2 step 9, `ARCHITECTURE.md`'s ASCII verifier flow, and `card_protocol_spec.md §7`'s summary all describe the same check — derive `keccak256(entry_pubkey)` for each `active_subcards` entry and confirm the sub-card's own address is present, as a hard rejection independent of the on-chain `SubCardEntry.active` flag (the two are treated as separate, both-must-pass checks, since they could in principle desync).

`SubCardDocument` gained a `limitations` field (`protocol-objects.md §16`), reusing the existing predicate/`field_requirements` grammar rather than a new vocabulary. `subcards.md §Limitations` documents two worked examples (a note-content size/character constraint addressing red-team Finding S-5, and a time-window constraint via timestamp-field regex) and is explicit that count-based rate limiting is out of scope for this stateless, per-message mechanism — a real limit, not hidden behavior.

`card_protocol_spec.md` now states explicitly (new subsection, "A Card's Schema Is a Floor, Not a Closed Allow-List") that a card's field set is required fields + declared policy fields + any additional undeclared fields, that undeclared fields are signed by all three parties but carry no synthesized `update_policy` (so they are effectively immutable after issuance — a known sharp edge, not hidden), and that §2 issuance-flow step 9's "schema satisfied" language means "required fields present and declared-field values valid," not a closed enumeration. `card_verifier.md` §7.5 step 2 carries the same clarification so the verifier package doesn't reject/choke on undeclared fields.

`subcards.md` also picked up two process additions implied by the new directory: a "Step 5b" describing the holder's separate code-510 post at sub-card issuance (distinct from the on-chain `RegisterSubCard` call), and a note that deregistration is paired with a code-511 removal. `press.md` documents both as separate updates from the on-chain calls it already handles. `client_sdk.md` documents the planned `resolveActiveSubCardTargets` helper (Step 4.1 of the implementation plan) for the "message all sub-cards" use case, and flags two **Not started** gaps in its Implementation Status table: the helper itself, and the code-510/511 posting the wallet SDK doesn't yet perform.

**Files touched:** `protocol-objects.md`, `update_codes.md`, `card_protocol_spec.md`, `subcards.md`, `ARCHITECTURE.md`, `process_specs/card_updates.md`, `process_specs/card_validation.md`, `process_specs/subcard_creation_policy.md`, `object_specs/card_verifier.md`, `object_specs/press.md`, `object_specs/client_sdk.md`. `object_specs/registry_contract.md` was reviewed and required no changes — it already treats `active_subcards` correctly as off-chain-only (the on-chain `SubCardEntry` doesn't store it), and rotation's on-chain-vs-off-chain shape is deferred to Phase 2 Step 2.1 as planned.

**No contradictions found** requiring resolution beyond the two additional forward-reference instances (`ARCHITECTURE.md`, `card_protocol_spec.md §7`) discovered during this consistency pass and fixed to match `protocol-objects.md §16`'s concrete language.
