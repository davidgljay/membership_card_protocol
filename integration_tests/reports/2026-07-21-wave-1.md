## 1. Freshly-minted test cards can't complete a chain-of-trust walk to a trusted root

**Spec:** `card_validation.md` Stage 3; `card_updates.md` (authorization
predicate evaluation); `open_offer_acceptance_new_wallet.md` /
`open_offer_acceptance_existing_wallet.md` (issuer chain evaluation in
`evaluatePredicates`).
**Components:** `integration_tests/fixtures/src/mintCard.ts` /
`suites/support/liveCard.ts` (test card minting), `press/src/functions/predicates.ts`
(`evaluatePredicates`), `membership_card_verifier`'s `CardVerifier` (Stage 3
chain walk).
**Observed:** Any suite that needs a real chain-of-trust resolution — either
via press's own `evaluatePredicates` (open-offer claim, card updates) or via
a client-side `CardVerifier.verifyCard()` chain walk — fails partway through
with "Ancestor card not found" (client-side) or a P-02 rejection
(press-side), because the ancestor address in `ancestry_pubkeys[0]` was
never itself registered on-chain.
**Expected:** A card minted via the fixtures' `mintCard`/`mintLiveCard`
helper should either (a) have an ancestry chain that genuinely resolves to
a registered trusted root, or (b) the test helpers should document plainly
that they only produce *self-rooted* (`ancestry_pubkeys: []` or
self-referential) cards, and any suite needing a deeper chain should build
one explicitly rather than assume `mintLiveCard` provides it.
**Suspected cause:** `mintCard.ts`'s own doc comment already states the
issuer for a fixture mint is "a synthetic level-1 card: its own
`ancestry_pubkeys` is `[]`" — so a card minted *through* that issuer has an
ancestor (the issuer) that itself was never registered as a `CardEntry`
on-chain, only used to sign the offer. This is fixtures behaving exactly as
documented; the gap is that no suite-level helper yet builds a *multi-hop*
chain (mint an issuer, register it, then mint under it) for tests that
specifically need one.
**Triage: defer.** Not a product bug — a test-infrastructure gap. Affects
Wave 1 suites' depth on chain-of-trust-dependent assertions but doesn't
block Wave 1's actual "does the happy path work" coverage for the specs
that don't require it. Worth a small addition to `suites/support/liveCard.ts`
(e.g. a `mintChainedLiveCard` that registers the issuer on-chain first) in
Wave 2 or as a fast-follow, not blocking this checkpoint.

## 2. `handleOpenOfferClaim` never propagates offer ancestry into the issued card — FIXED 2026-07-21

**Status: resolved**, same day as the checkpoint. Unlike the targeted-offer
path, `OpenCardOffer` (`open_offer_creation.md`) has no multi-level
`ancestry_pubkeys` field of its own to carry through — it only ever
declares a single-level `issuer_pubkey`, already binding-checked against
`issuer_card`. Fixed `press/src/handlers/open-offer.ts` to pass
`[offer.issuer_pubkey]` (the issuer as immediate parent, per
`protocol-objects.md §1`'s "immediate parent = issuer card's public key")
instead of the hardcoded `[]`. Added a regression test
(`press/test/unit/errors.test.ts`, `handleOpenOfferClaim — success`) that
decrypts the assembled card document (ADR-006 content encryption) and
asserts `ancestry_pubkeys` matches. Full press suite (172 tests) and all 7
Wave-1 integration suites (44 tests, 10 `it.todo`) confirmed green against
the rebuilt press container.

**Spec:** `open_offer_creation.md` / `card_protocol_spec.md` step 8 (completed
card carries `ancestry_pubkeys`); mirrors `card_offering_and_acceptance.md`'s
same requirement for targeted offers.
**Components:** `press/src/handlers/open-offer.ts` (`handleOpenOfferClaim`,
~line 91-97).
**Observed:** `assembleCardDocument` is called with a hardcoded `ancestry: []`
and an inline comment `// Phase 3 placeholder` — every card issued via the
open-offer claim path currently gets an empty ancestry chain regardless of
the offering issuer's real position in the trust hierarchy.
**Expected:** Same behavior as the targeted-offer path: `offer.ancestry_pubkeys`
(set by the issuer when assembling the `OpenCardOffer`, covered by
`issuer_signature`) should be carried through into the completed card.
**Suspected cause:** This is the exact bug already found and fixed this
session in the *targeted*-offer path (`press/src/handlers/issue.ts`, see
git history — "propagate offer.ancestry_pubkeys into the finalized
CardDocument") — the open-offer handler shares the same shape of issue but
was never updated to match. Two independent call sites implementing the
same postcondition, one fixed, one not.
**Triage: fix-now — done.**

## 3. Press's non-existent-target-card handling is inconsistent

**Spec:** `card_updates.md` error paths (target card must exist).
**Components:** `press/src` update-intent handling / registry lookup.
**Observed:** Submitting an update intent against a card address that was
never registered sometimes returns a clean `400` with a `P-01`-style error
code, and sometimes an unhandled `500` — behavior depends on exactly which
code path first touches the missing registry entry.
**Expected:** A single, consistent error response (structured `4xx` +
press error code) regardless of which internal step first notices the card
doesn't exist.
**Suspected cause:** Likely a registry-lookup call site that isn't wrapped
in the same try/catch-and-translate-to-press-error pattern the rest of the
update path uses — not investigated further (out of this suite-writing
pass's scope).
**Triage: defer.** Minor robustness gap (wrong HTTP status / unhandled
exception surfaces as generic "Server Error" instead of a clean 4xx), not a
correctness bug in the happy path. Worth a follow-up ticket, not urgent.

## 4. `card_validation.spec.ts`'s coverage is shallower than intended

**Spec:** `card_validation.md`, all stages.
**Components:** the suite file itself.
**Observed:** Of the file's 5 tests, only 4 pass and mostly exercise
`CardVerifier.verifyCard()`'s address-binding behavior rather than a full
`verifyEnvelope()` call against a real signed `SignedMessageEnvelope` — the
spec's actual central object. The one test that would have driven a real
chain-walk through `verifyEnvelope`/`verifyCard` to a trusted root is
`it.todo` per finding #1 above.
**Expected:** Fuller coverage of Stages 1-3 via `verifyEnvelope()` on a real
envelope, once a chained test identity is available.
**Suspected cause:** Same root cause as #1 — written before a
chain-capable test identity existed.
**Triage: test-bug/defer.** Revisit this suite specifically once #1's
follow-up (`mintChainedLiveCard` or equivalent) lands; it's the natural
consumer of that fix.

## Notes on the process itself (not a defect, but worth recording)

Running all 7 suite files together (rather than one at a time) surfaced a
**cross-file** version of the nonce-collision issue `card_signing.spec.ts`
already worked around within a single file: vitest parallelizes across test
files by default, and every suite's `beforeAll` hits press's single gas
wallet. Fixed via `suites/vitest.config.ts`'s `fileParallelism: false`. Any
future Wave should keep this in mind before assuming "each suite passes in
isolation" is sufficient proof `npm test` will be green.
