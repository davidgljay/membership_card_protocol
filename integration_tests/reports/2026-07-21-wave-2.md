## 2. No Application Service wired into this stack's Synapse

**Spec:** all of Wave 2 — every Matrix suite needs a real Matrix account
for a card.
**Components:** `integration_tests/env/synapse/homeserver.yaml.template`,
`wallet-service/src/matrix/{provisioning,token-minting,appservice-tokens}.ts`.
**Observed:** `wallet-service`'s `POST /matrix/token`/`POST /matrix/rooms`
(the "normal" way a client provisions a shadow Matrix account and creates
a card-gated room) fail before ever reaching Synapse — `readAppServiceAsToken()`
reads a file (`wallet-service/matrix/secrets/appservice-as-token.txt`)
that doesn't exist in this stack's `wallet-service` image, and this
stack's Synapse has no `app_service_config_files` entry to accept an AS
login even if a token were supplied. The `homeserver.yaml.template`'s own
header comment already documents this as a known, deferred TODO — not a
surprise found this session.
**Expected:** An Application Service registered for wallet-service's
shadow-account bridge, matching `wallet-service/docker-compose.yml`'s own
(separate) stack, which does have this wired.
**Suspected cause:** Genuinely just not done yet — Task 1.7 ("wallet-service
under workerd") brought wallet-service into this compose stack but never
closed this specific loop.
**Workaround used, not a fix:** every Wave-2 suite bypasses wallet-service's
provisioning entirely (`suites/support/matrixAdmin.ts`) — registering test
Matrix users directly via Synapse's admin shared-secret API at the exact
`@card_<hex>:server` localpart a real shadow account would have.
`deriveMatrixUserId`/`verifyMatrixUserIdBinding` are pure functions of a
card's keypair, so this is a full substitute for everything the *policy
module* cares about — but it means nothing in Wave 2 exercises
wallet-service's own AS-integration code at all, and `room_discovery.spec.ts`'s
room index stays permanently empty in this environment (§1's
`POST /matrix/rooms` write path is unreachable), since the bypass creates
rooms directly on Synapse, never touching wallet-service's Postgres.
**Triage: defer.** Real, bounded, has a working reference implementation
to copy from (`wallet-service/docker-compose.yml`'s own AS wiring) — but
non-trivial (secrets generation/mounting, `app_service_config_files`,
confirming `wallet-service`'s file-based token-reading approach even
survives its `cloudflare-module`/workerd build target, which is genuinely
uncertain and untested). Worth scoping as its own task rather than folding
into Wave 3.

## 3. Chain-walk path crashes under Synapse's Twisted reactor — confirmed live, not fixed

**Spec:** `matrix_join_attestation_and_revocation.md §3.3`'s "Predicate
evaluation itself throws... Deny... This is the module's last line of
defense — a bug in the evaluator must not become an accidental allow."
**Components:** `matrix_policy_module/attestation.py`'s
`verify_join_attestation` → `chain_context.py`'s
`walk_join_attestation_chain` → `membership_card_verifier`'s
`CardVerifier.verify_envelope` (Python package, `card_verifier.py`).
**Observed:** A validly-signed, correctly-bound join attestation for a
card that was never minted anywhere (no chain to walk) reaches real
chain-walk code and crashes with `RuntimeError: await wasn't used with
future` inside `card_verifier.py`'s `asyncio.gather(...)` call — a
Twisted-reactor/asyncio-Future mismatch somewhere in the RPC/IPFS provider
bridging `chain_context.py` hands to the verifier. The exception is not
caught; it propagates to Synapse's HTTP layer as a raw `500`, not the
`403 M_FORBIDDEN` every other deny path produces.
**Expected:** A `403` deny, per the spec's own explicit "last line of
defense" guarantee — a `500` is not an allow, but it's also not the
guarantee the spec promises.
**Suspected cause:** Not investigated further — this is genuinely
Python-async-runtime-integration work (understanding how `chain_context.py`'s
RPC/IPFS provider bridges Twisted's Deferred-based reactor to
`asyncio.gather`), not something to rush mid-suite-write. Full traceback
captured in `matrix_room_membership.spec.ts`'s own comment.
**Triage: fix-now candidate, but scope it properly rather than rush it.**
This is the first attestation in the whole suite set to actually reach
real chain-walk code — nothing in Wave 1 or the rest of Wave 2 exercises
this path, so it's plausible this bug has been live and undetected since
whenever `chain_context.py`'s provider bridging was last touched. Test
suite captures the current (buggy) behavior via a regression trip-wire
assertion (`matrix_room_membership.spec.ts`) so a silent regression to an
actual *allow* would be caught, and an `it.todo` for the correct behavior.

## 4. Room creator's own first post is denied — confirmed live, not fixed

**Spec:** `matrix_join_attestation_and_revocation.md §2`'s "Creator
auto-join, carried over unchanged" — states plainly that the creator's
membership "must still be entered into the membership registry... or
their first post would have no registry entry to resolve against."
**Components:** `wallet-service/src/matrix/room-creation.ts`'s
`createMatrixRoomViaSynapse`.
**Observed:** Confirmed via a throwaway probe script before writing the
formal test, then encoded as a real regression assertion
(`matrix_join_attestation_and_revocation.spec.ts`): a room creator's very
first `PUT .../send/m.room.message/...` to their own freshly-created room
is denied `403 M_FORBIDDEN`, because nothing in the room-creation flow
calls into the membership registry to record `(room_id, creatorMatrixUserId) →
card_hash` for the auto-join Synapse itself never routes through
`check_event_allowed`.
**Expected:** The creator's auto-join registered directly by whatever code
handles room creation, per the spec's own explicit instruction.
**Suspected cause:** `createMatrixRoomViaSynapse` only calls Synapse's
`/createRoom` and returns — the membership-registry write the spec calls
for was never added.
**Triage: fix-now candidate.** Confirmed, understood, and the fix is
localized to `room-creation.ts` (a call into the same membership-registry
mechanism `check_event_allowed`'s join path already populates on every
non-creator join) — smaller in scope than item 3 above. Worth doing
alongside or instead of item 3 if only one can be prioritized before
Wave 3.

## 5. `wallet-service-postgres` has no host port mapping

**Spec:** N/A — test-infrastructure limitation, not a spec compliance
issue.
**Components:** `integration_tests/docker-compose.yml`'s
`wallet-service-postgres` service.
**Observed:** `wallet-service/server/db/matrix-rooms.ts`'s
`insertRoomIndexEntry` is a plain, directly-callable function that could
seed real `room_discovery.spec.ts` fixtures without going through the
AS-blocked `POST /matrix/rooms` path — but the suite (running on the
host) can't reach the database to call it, since this service isn't
published to the host like `synapse-postgres`'s peers are elsewhere in
this file.
**Triage: defer.** A one-line `ports:` addition, low risk, but not done
opportunistically mid-suite — would need to confirm it doesn't
conflict with `wallet-service`'s own port 3002 mapping or introduce a
stray host-exposed credential surface worth a moment's thought, not a
rushed one-liner.

## 6. Single-wallet-service topology limits `message_routing.md` coverage

**Spec:** `message_routing.md`'s "sender WS A → recipient WS B" scenario.
**Observed:** This stack runs one `wallet-service` instance, so true
inter-wallet-service routing (an announcement lookup resolving to a
*different* wallet service's endpoint) isn't literally exercisable.
Self-routing (one wallet service routing between two cards it holds) is
tested instead.
**Triage: defer / accept as environment scope, not a defect.** Matches
this session's established pattern (Wave 1's similar single-instance
scoping notes) — spinning up a second wallet-service instance for one
suite's sake isn't proportionate.
